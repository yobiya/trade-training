## 1. 概要

### 1.1 目的
チャート分析・エントリー判断・エリオット波動分析のスキルを実戦的に訓練するための個人用アプリ。

### 1.2 コンセプト
- **時間起点の銘柄選定**: 特定の日時を先に提示し、その時点で優位性のある銘柄をユーザー自身が選ぶ
- **トレードスタイル選択**: 短期・中期・指標狙いなど複数のスタイルを状況に応じて使い分ける訓練
- **シナリオメモ主導**: 思考プロセスを必ず言語化し、セッション内で振り返り可能にする
- **見送りも評価**: 「入らなかった判断」も §9.3 の機械判定で事後評価
- **経済指標の認識訓練**: 強指標をチャート上に可視化、相場の背景要因を意識した判断を促す
- **トレーニングとリアルの一貫性**: 同じ分析・メモ機能を実トレードでも使用可能
- **AI 分析支援**: 1 セッション単位の深掘り観察(§11)。集計・情報蓄積は採用しない([§10](./03-analysis.md#10-集計情報蓄積を採用しない方針))

### 1.3 対象ユーザー
自分専用(単一ユーザー)

### 1.4 プラットフォーム
Web + スマホ両対応(レスポンシブデザイン)

### 1.5 システム構成(コンポーネント分割)

本システムは**2 つの独立アプリ + 3 つの共通パッケージ**で構成し、全てを**Windows VPS(XServer VPS等)一台で稼働**させる。モノリシックアプリではなく責務ごとに分割することで、開発の段階化・リスクの局所化・再利用性を確保する。

```
[クライアント]
  スマホ / PC のWebブラウザ
     ↓ HTTPS
[Windows VPS (24/365稼働)]
  ├─ MT5本体 (常時起動、RDPで管理)
  │
  ├─ [共通パッケージ]
  │    shared-schema        ─ DB接続・スキーマ定義・マイグレーション
  │    common-ui-lib        ─ チャート/メモフォーム/描画ツール等の共有UI
  │    market-data          ─ 市場データ取得ライブラリ(ハイブリッドキャッシュ)
  │
  ├─ [共有ストア]
  │    SQLite (trading.db)  ─ 全アプリが参照する単一ファイル
  │
  ├─ [アプリ B] trade-trainer (Webサーバー)
  │    └ 過去チャートを使った訓練。発注コードを一切含まない
  │      AI 分析(§11)は本体機能として統合(別アプリ化しない)
  │
  └─ [アプリ C] trade-live (Webサーバー)
       └ 実トレードの実行。MT5経由で発注、ポジション管理
     
     ↓ MT5 Python API (同一マシン内、遅延数ms)
  [ブローカーサーバー]
```

横断集計アプリ(旧称 trade-analyzer)は採用しない([§10](./03-analysis.md#10-集計情報蓄積を採用しない方針))。集計・統計機能を持たず、分析は §11 AI 分析(1 セッション単位)で完結させる。

**市場データ取得の方針: ハイブリッドキャッシュ**

価格データは**事前バルク取得せず**、使われた範囲を自動的にDBキャッシュする遅延ロード方式を採る。

```
[アプリが価格データを要求]
     ↓
[market-data ライブラリ]
  ├─ SQLiteキャッシュを確認
  │   ├─ ある → キャッシュから返す(高速)
  │   └─ ない → MT5から取得 → SQLiteに保存 → 返す
  ↓
[アプリ]
```

- **初回セットアップ不要**: 使いながら自動的にキャッシュが溜まる
- **MT5依存は初回のみ**: 一度使った範囲はMT5が落ちていても利用可能
- **経済指標データは事前取得**: 価格と異なり量が少なく、日次バッチ更新で済む(market-dataのCLIで実行)

**分割の基本方針**
- **全アプリがVPS内で稼働**: MT5・アプリ・DBが同居、発注遅延がほぼゼロ
- **発注コードの隔離**: 誤発注防止のため B には発注コードを一切含めない。発注機能は C のみ
- **UIの共通化**: チャート・メモ・描画ツール等は `common-ui-lib` として B/C で共有
- **データ取得の共通化**: 市場データ取得は `market-data` ライブラリに集約、B/C から呼び出し
- **データ単一化**: 2 アプリ全てが同じSQLiteファイルを参照。スキーマは `shared-schema` で統一管理
- **モバイル完全対応**: クライアントはブラウザのみ、スマホからHTTPS経由で全機能利用可能

**各コンポーネントの関係**

| コンポーネント | 種別 | 実行環境 | DB操作 | MT5接続 | 発注機能 | 開発Phase |
|---|---|---|---|---|---|---|
| shared-schema | パッケージ | - | - | - | - | Phase 1 |
| common-ui-lib | パッケージ | - | - | - | - | Phase 2 |
| market-data | パッケージ | - | 読み書き(市場データキャッシュ) | 取得 | - | Phase 1 |
| B. trade-trainer | Webアプリ | VPS | 読み書き(トレーニング記録) | 経由(market-data 経由) | **なし** | Phase 1〜4 |
| C. trade-live | Webアプリ | VPS | 読み書き(リアル記録) | 直接(発注) + 経由(データ) | あり | Phase 5〜6 |

※ 集計専用アプリは採用しない([§10](./03-analysis.md#10-集計情報蓄積を採用しない方針))。AI 分析(§11)は B 本体に統合する。

**開発環境と本番環境**
- 本番: Windows VPS
- 開発: 手元PCでもアプリBは動作可能(MT5非依存、market-dataはキャッシュだけ参照するモードで起動)。アプリC および market-data の MT5取得パスはVPS上で開発・テスト

**DBスキーマ管理**
`shared-schema` パッケージに以下を持たせる:
- SQLAlchemy によるスキーマ定義
- alembic マイグレーションスクリプト
- 2 アプリ + market-data がこのパッケージを依存関係として使う

スキーマ変更時は `shared-schema` を更新し、全アプリをそれに追従させる。

## 2. データ基盤

### 2.1 データソース
- **初期実装**: MT5 (`MetaTrader5` Python パッケージ)
- **将来拡張候補**(MT5で必要データが取得できない場合): Dukascopy, HistData, TrueFX 等
- **market-data** パッケージが**データソース抽象化**されており、設定ファイルで切り替え可能

### 2.2 保存データ(市場データ)
- **ハイブリッドキャッシュ方式**: 事前バルク取得は行わず、使われた範囲を自動的にSQLiteにキャッシュ
- **M5(5分足)を最小単位として管理**
- ユーザーのトレードスタイルの最小時間軸がM5のため、M1は保存しない
- 上位足(M15/M30/H1/H4/D1等)は動的集約で生成
- **最小保存足は設定可能**: 将来スキャルピング訓練等でM1が必要になった場合は設定変更で対応
- **DB**: SQLite(WALモードで読み書きの並行性が良い。2 アプリでの共有に適する)
- **データの扱い**: キャッシュレコードには取得日時・データソースを記録、後から再取得判定が可能

### 2.3 保存データ規模の見積もり
- 使われた範囲のみキャッシュする方式のため、初期は小さく徐々に増える
- 市場データキャッシュの全期間蓄積上限: M5 で 5 年分 1 銘柄あたり約 40〜60 MB(これは [§10](./03-analysis.md#10-集計情報蓄積を採用しない方針) の対象外。市場データは毎回同じ日時を訓練に使っても取得が必要で、キャッシュしないと MT5 負荷が上がるため保持する)
- 10銘柄でも最大400〜600 MB、30銘柄でも2 GB未満
- SQLiteの上限(理論上 281 TB)から見て余裕、実用上問題なし

### 2.4 データソース抽象化の設計
全てのデータソースは共通インターフェースを実装し、差し替え可能とする。

```python
class DataSourceProvider:
    def fetch_ohlc(symbol, timeframe, from_dt, to_dt) -> DataFrame: ...
    def fetch_latest(symbol, timeframe) -> DataFrame: ...
    def get_available_range(symbol) -> (from_dt, to_dt): ...
```

**正規化ルール**(プロバイダー内部で吸収)
- タイムゾーン: **データ保存は全てUTC**に統一(表示は後述のUI層でJSTに変換)
- 価格基準: 全て**Bid基準**に統一
- 銘柄名: 接尾辞を除去した統一形式(例: `USDJPY.a` → `USDJPY`)
- 出来高: ティック数または出来高(プロバイダー差異はメタ情報で管理)

### 2.5 DBスキーマでのソース管理
複数ソース共存を可能にするため、`source` カラムを設ける。

```sql
CREATE TABLE ohlc_m5 (
  symbol TEXT,
  timestamp TIMESTAMP,
  open DOUBLE, high DOUBLE, low DOUBLE, close DOUBLE,
  volume BIGINT,
  source TEXT,  -- 'mt5', 'dukascopy', etc.
  PRIMARY KEY (symbol, timestamp, source)
);
```

これにより、途中でソースを切り替えても過去データを活かせる。

### 2.6 トレード記録との整合性
データソース変更時に既存のトレード記録が影響を受けないよう、以下を守る:
- エントリー時点の価格は**スナップショットとしてトレード記録に保存**
- スプレッドは**銘柄別固定値**を使用(ソース依存しない)

### 2.7 取得対象期間
- **想定レンジ**: 過去5年程度までを訓練対象とする
- 事前バルク取得はしないため、ユーザーがトレーニング中に選んだ期間だけが段階的にキャッシュされる
- MT5が保持しているデータ範囲はブローカー依存のため、Phase 1 で主要銘柄の過去5年分が取得可能かを**検証**する
- 取得困難な期間がある場合は Dukascopy 等への切り替えを検討

### 2.8 対象銘柄
- 設定画面でユーザーが指定
- 有効化した銘柄群を出題対象として使用
- **初期対象(デフォルト8ペア)**: `USDJPY`, `EURUSD`, `GBPUSD`, `AUDUSD`, `EURJPY`, `GBPJPY`, `AUDJPY`, `EURGBP`
  - メジャー通貨ペア + 主要クロス円を網羅
  - ユーザーは設定画面から追加・無効化可能

### 2.9 データ更新
- **価格データ**: ユーザーのアクセス時に market-data ライブラリが自動取得・キャッシュ(明示的な更新ボタンは不要)
- **経済指標データ**: Windowsタスクスケジューラで market-data CLI を日次実行、新しい指標予定を取り込む
- **手動更新**: アプリUIから market-data CLI を呼び出す機能も提供(設定画面等)

### 2.10 経済指標データ
- **第一候補**: MT5 Economic Calendar API (`calendar_value_history()` 等)
- **代替**: 外部CSVの事前取り込み(ブローカーのMT5環境で取れない場合)
- **取得方式**: 事前バッチ取得(market-data CLIの日次実行)。価格と違いデータ量が少なく、予定値を含めて事前に知りたいため
- **保存内容**: 発表日時、通貨、指標名、重要度、実測値、予想値、前回値
- **重要度扱い**: MT5の `Importance` フィールドを使用(3段階)
- **時刻精度の注意**: 最小保存足がM5のため、指標発表時刻を含むM5足が1単位となる。発表前後の細かい1〜2分単位の動きは確認できないが、ボラの塊としては十分可視化される

### 2.11 タイムゾーン表示ポリシー
- **保存層**: 全データを**UTC**で保存・処理(DB・API・バックエンド)
- **表示層**: ユーザー向け表示は全て**JST(Asia/Tokyo)**に変換
- **変換責務**: フロントエンド側で変換(`dayjs` + `dayjs/plugin/timezone` または `date-fns-tz` 等)
- **変換対象**:
  - チャート時刻軸・クロスヘア・ツールチップ
  - 経済指標の発表時刻表示
  - セッション日時・トレード記録の日時表示
  - シナリオメモの入力時刻・保存時刻
- **入力層**: 時間フィルタ等の時刻入力もJSTで解釈(入力UIに「(JST)」と明記)
- **シナリオメモ本文中の時刻記述**: ユーザーは時刻を**JSTで記述**する前提。例「23時のFOMC前に利確」「翌朝9時に再評価」。システムは本文内の時刻を自動変換しない。入力UIに「時刻はJST」とプレースホルダー表示。AI分析にメモを渡す際もプロンプトに「メモ内時刻はJST基準」と明記する
- **セッションプリセット**(JST基準、夏時間は考慮せず固定):
  - 東京: 09:00〜15:00
  - ロンドン: 16:00〜25:00
  - NY: 22:00〜翌06:00

## 3. 取引条件

| 項目 | 扱い |
|---|---|
| スプレッド | 銘柄別固定値を設定ファイル (`config/spreads.toml`) で管理 |
| スリッページ | なし |
| スワップ | なし |
| ロット・証拠金 | なし(pips損益ベースで評価) |

**スプレッド初期値の確定方針**
- MT5 デモ口座接続前は**暫定値**を `config/spreads.toml` に置く(一般的な値で運用)
- MT5 デモ口座の接続が通った時点で、実測値(平均スプレッド)に更新
- 設定画面からもユーザーが編集可能(Setting テーブル経由)

## 4.1 通常セッション(統合フロー・1 画面完結)

画面分割はしない。「銘柄選定」「トレーニング」と段階分けするのではなく、**1 つのセッション画面** の中で、分析 → エントリー(= そこで初めて銘柄が確定する) → 保有 → 決済 → 振り返り が流れる。

```
[Phase 1: 時間選定]
1. 時間フィルタ設定(セッション・曜日・期間など)
2. システムがランダムな日時を提示してセッション開始(銘柄はまだ決まっていない)
   - 提示時刻は **JST 08:00 から翌 JST 02:00 まで**(18 時間枠)に限定する。JST 02:00〜08:00 の「東京クローズ後・ロンドン前」は流動性が極端に低く訓練価値が低いため除外する(`_matches_filters` で実装)

[Phase 2: 分析とエントリー判断(1 画面内で完結)]
3. 銘柄リストから気になる銘柄を選んで詳細チャート(マルチタイムフレーム)で観察
4. 描画ツール・インジケーターで分析(サポレジ・トレンドライン・波動カウント 等)
5. メモパネル(§7.3、M キー or ボタン)を開閉しながら、銘柄別メモに個別の気づき、
   横断メモに通貨強弱・銘柄比較・相場観・シナリオを書く
6. 現時点がエントリーに適さなければ ▶ で足を進めて待つ。環境が変わったら分析をやり直す
7. 比較のために別銘柄に切り替えてもよい(描画は銘柄別に保持され、切替で自動的に該当銘柄の描画に切り替わる)
8. ★ で気になる銘柄をブックマーク(任意、銘柄別メモの入れ物として機能)
9. 条件が揃った銘柄で **SL(必須)・TP(任意)をチャート上に配置** してエントリー
   - エントリー価格は **現在値で自動固定**(成り行き想定)
   - 方向は **SL の位置から自動判定**(SL < 現在値 → BUY、SL > 現在値 → SELL)
   - 配置後に表示される単一の「エントリー」ボタンで確定
   または、優位性なしと判断したら「見送り」
   ※ エントリーアクションそのものが「この銘柄でこのセッションをやる」という選定確定
   ※ 必要な思考(根拠・シナリオ)は銘柄別メモ・横断メモに事前記入済みの前提

[Phase 3: 保有中(エントリーした場合のみ)]
10. 銘柄は Phase 2 のエントリーで確定。以降は切替できない
11. ▶ で足を進めて TP/SL 到達 or 裁量決済まで進む。必要なら描画を追加、メモに気づき追記
12. 決済(自動 or 裁量)

[Phase 4: 振り返り]
13. 決済後の続き観察(追加 N 本の OHLC チャート + MFE・MAE・実損益 R を R 単位で確認、[§9.5](./09-post-review.md#95-エントリー結果の事後確認))
14. 見送り時は事後チャートと [§9.3](./09-post-review.md#93-事後評価の表示方針) の最大上昇 R / 最大下落 R(pips 補助)を PostReviewPanel で確認(機械判定ラベルは採用しない)
15. 横断メモに振り返り追記。決済済み or 見送り確定後の最初の note 更新で **決着済みに自動遷移**([§4.2.2](#422-決着遷移条件))。後日追記も可([principles/no-aggregation.md#103-セッションのライフサイクル](./principles/no-aggregation.md#103-セッションのライフサイクル))
16. セッションは決着済みのまま `data/sessions/{dir}/` に永続化される。削除は OS / Dropbox 上のディレクトリ操作で行う([§13](./13-data-storage.md))
```

## 4.2 セッション管理
- 時間制限なし
- セッションはファイルベースで永続化([§13](./13-data-storage.md) / [§17](./17-data-model.md))。アプリで自動破棄せず、削除はユーザーが OS / Dropbox 上のディレクトリ操作で行う
- **1セッション1エントリー完結**(trainer は単一判断の訓練に特化)。決着済みになっても再エントリーは不可
- 複数ポジション管理(資金配分・優先順位判断・ヘッジ・段階エントリー)は trainer の対象外。これらは実戦環境(journal)で経験として学ぶ方針
- trainer で身につけるのは「1つの局面に対するエントリー・決済判断の質」。実戦で必要な「同時に複数のポジションをどう捌くか」は journal 側で扱う
- **Phase の行き来を許容する**: §4.1 のフローは概念的な順序であり、実際の訓練では Phase 2 の中で「銘柄切替 ↔ 分析 ↔ 足送り ↔ メモ」を自由に往復できる。Phase 3 以降(エントリー後)は銘柄が確定するが、描画・メモ追記は継続可能。
- **画面遷移は一度もない**: 分析・エントリー・保有・決済・振り返りはすべて同じ `SessionPage` 内で行い、セッションの状態(`settled_at` の有無 / アクティブ Trade の有無 / skip 済か)からフェーズを導出して UI 要素を出し分ける。
- **セッション名はいつでも編集可**: 振り返り時に手法を識別する手がかりとして、`Session.name`(任意・自由記述)を分析中・保有中・振り返り中のどのフェーズでも編集できる。手法が見えた段階で書き残せばよい。

### 4.2.1 セッション状態モデル(進行中 / 決着済み)

セッションは「**進行中** (active) / **決着済み** (settled)」の 2 状態を持つ。`session.json` の `settled_at: ISO8601 | null` フィールドで管理する。

- **進行中** (`settled_at == null`): 振り返りメモがまだ書かれていない状態。トレード未決済も、決済済みだが振り返り未着手も含む
- **決着済み** (`settled_at != null`): ユーザーが自分の中で一区切りつけた状態。中身が中途半端でも振り返りメモが書かれていれば該当

「決着済み」は「終わり・破棄」ではなく、**結果が出てユーザーが振り返りに着手したマイルストーン**を意味する。決着済みになってもメモ・描画の追記・編集は継続可能で、後日の振り返りで内容を更新してよい。

### 4.2.2 決着遷移条件

以下の **両方** が満たされた時点で、自動的に `settled_at` がセットされる:

1. **トレード決済済み** (`trade.exit_time` あり)、または **見送り確定済み** (`final_decision.has_entry == false`)
2. **横断メモ `note.md` (Session.note) が更新された**(空文字以外、中身は中途半端でも可)

判定対象は **横断メモのみ**。銘柄別メモ(`candidates/{symbol}.md`)は判断中・保有中にも書かれるため判定トリガーには含めない。逆遷移はなし(決着済みから進行中には戻らない)。

「保留中」を表す独立フラグは設けない。決着済みになっても編集は引き続き可能なので、振り返りメモを後日に書く目的で保留状態を別建てで持つ必要がない。

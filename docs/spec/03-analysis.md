## 9. 見送り検証機能

### 9.1 2層の見送り記録

**層1: 候補外し** - ウォッチリストに入れたが最終選定しなかったペア
**層2: エントリー見送り** - 最終候補まで絞ったが「入らない」と判断したケース

### 9.2 事後データ表示
セッション終了時、見送ったペアについて以下を表示:
- **3段階の事後チャート**(10本先 / 50本先 / 200本先)
- 各期間の最大上昇幅・最大下落幅
- 買い/売り想定時の推定成績
- 実際にエントリーしたペアとの比較

### 9.3 自己評価タグ(ユーザー入力)

**層2用タグ(必須)**
- `正解_ノーエントリー`: 入らなくて正解
- `機会損失_エントリー`: 入るべきだった
- `正解_方向判断`: 見送り理由は妥当
- `慎重すぎ`: 過度に慎重だった

**層1用タグ(任意・スキップ可)**
- `正解`: 選ばなくて良かった
- `機会損失`: 選ぶべきだった
- `判断通り`: 選定基準的に外して正しい
- `どちらでも`: 差はなかった

### 9.4 集計指標

**見送り単体指標**
- 見送り正解率
- 機会損失率
- 機会損失の平均逸失pips
- 慎重すぎ指数

**組み合わせ指標**
- 選定精度(エントリー vs 見送り、動きが大きかったのはどちらか)
- 逆選定率
- **総合トレード判断スコア** = (エントリー成功 + 見送り正解) ÷ 総判断数

## 10. 成績評価指標

### 10.1 基本指標
- 勝率
- 平均獲得pips / 平均損失pips
- 実績リスクリワード比
- プロフィットファクター(総獲得pips ÷ 総損失pips)
- 期待値(pips/トレード)
- 最大連勝・最大連敗

### 10.2 後悔指標
- 最大利益取得率(実際の獲得 ÷ 持ち続けた場合の最大獲得)
- 最大逆行許容度
- 「続き観察」期間の追加動向

### 10.3 多次元集計
- タグ別成績
- シナリオ別成績
- 時間軸別成績
- 銘柄別成績
- **スタイル別成績**(短期/中期/指標/スイング等、選択スタイルごとの勝率・期待値)
- **スタイル × 時間帯**(短期スタイルはNY時間で勝率高い等、組み合わせ分析)
- **スタイル × 指標絡み**(指標スタイルの発表前後成績、短期スタイルの指標回避妥当性)
- **スタイル想定とのズレ**(想定保有時間・想定RRと実績の比較、スタイル定義の見直し材料)
- **セッション別勝率**(東京/ロンドン/NY/閑散)
- **曜日別勝率**
- **指標絡み成績**(指標前◯分 / 指標後◯分 / 指標無風時の勝率比較)
- シナリオ的中率(記入したシナリオのうちどれかに当たった率、記入数も併記)
- 選定候補数別勝率

### 10.4 学習継続指標
- 連続学習日数
- 総セッション数
- 週次・月次レポート(自動生成)

## 11. AI分析機能

### 11.1 実装方式: ハイブリッド
- **定量分析**: ローカルPython(集計・統計・グラフ生成)
- **定性分析**: Claude API(メモ解釈・コーチングレポート生成)

### 11.2 提供場面

ダッシュボードから「AI分析」ボタンで起動する**オンデマンド分析のみ**提供する。セッション直後の自動分析は行わない(コスト抑制・ノイズ回避のため)。

- **範囲選択**: 直近10トレード / 過去1ヶ月 / 全期間 / カスタム期間
- **モード選択**: training / real / 横断
- **観点選択**: 総合 / 勝ちパターン / 見送り精度 / メモ品質 / スタイル選択

### 11.3 分析観点

**トレード傾向**
- 勝ちパターン・負けパターンの発見
- シナリオメモのキーワード × 勝率の相関
- 曖昧な表現(「なんとなく」など)の検出

**見送り判断の質**
- 見送り理由別の正解率
- 機会損失パターンの特定

**シナリオ的中分析**
- 3シナリオ事前宣言の的中率推移
- 想定外パターンの傾向

**スタイル選択の質**
- スタイル選定理由と実績の相関
- 選んだスタイル vs 他スタイルでの仮想成績比較
- スタイル定義の想定(保有時間・RR)と実績のズレ
- 局面に対するスタイル選択のクセ

**成長分析**
- 期間別の成績推移
- メモの質的変化
- スランプ検出

### 11.4 データ送信とコスト・プライバシー管理

**送信前プレビュー(必須ゲート)**
- 分析実行前に「送信予定データ」のプレビュー画面を表示
- ユーザーが確認して「送信」ボタンを押して初めて API 実行
- プレビュー上でチェックボックスにより個別トレード・メモの**送信除外**が可能
- 一度送信したデータは取り戻せないため、実行前の確認を必須化

**バジェット管理**
- 設定画面で月額バジェット上限を指定(デフォルト: 2000円相当)
- 実行前に推定トークン数・推定コストを表示
- 上限に達している場合は実行不可(翌月リセット)
- 利用量ダッシュボード: 当月/先月の実行回数・送信トークン数・推定コストを可視化

**キャッシュ**
- 入力データのハッシュでキャッシュ
- 同一入力の再分析はキャッシュから返す(API コールしない)
- ユーザーが明示的に「再実行」を選んだ場合のみ再コール

**時刻情報の扱い**
- 送信データ内の日時は ISO8601(UTC)で送信
- システムプロンプトで「ユーザーは JST で運用しており、メモ本文内の時刻は JST 基準」と明記し、AI が誤解釈しないようにする

### 11.5 分析コンテキストの導出方針

AI 送信時に必要な市場コンテキストは、**DBへの冗長保存は行わず、`entry_time`/`exit_time` を起点に on-demand で導出**する方針とする。

| 導出項目 | 導出元 | 実装場所 |
|---|---|---|
| エントリー/決済時点のチャート(各時間足 N 本) | `ohlc_m5` + `market-data` のリサンプル | `market-data.accessor.get_ohlc()` |
| MAE/MFE(最大含み損・含み益) | entry_time〜exit_time の M5 OHLC から min/max 計算 | `market-data` にヘルパー追加予定 |
| 近接する経済指標 | `economic_event` を entry_time 前後 ±N時間で検索 | クエリ関数 |
| セッション時間帯(東京/ロンドン/NY) | entry_time の UTC 時刻から判定 | ユーティリティ関数 |
| ATR / スイングハイ・ロー | 直近 N 本の OHLC から計算 | ユーティリティ関数 |
| インジケーター値(SMA/EMA/RSI 等) | OHLC + 当時の設定から再計算 | 下記参照 |

**例外:インジケーター設定スナップショット**

ユーザーが後から期間・種類設定を変更すると、過去トレード分析時に当時の値を再現できない。対策として:

- `Setting` にインジケーター設定リストを保持し、変更時に**新バージョンを追記(履歴保持)**する
- `TradeSession` に `indicator_config_id`(その時点の設定バージョン ID)を紐付け
- 分析時はこの ID を介して当時の設定を取得し、OHLC から値を再計算

この方式により、描画オブジェクト・メモ等の**ユーザー入力のみを保存**し、市場コンテキストは導出で再現する設計を貫徹する。

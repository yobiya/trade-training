## 9. 見送り検証機能

### 9.1 2層の見送り記録

**層1: 候補外し** - ウォッチリストに入れたが最終選定しなかったペア
- 入力タイミング: [§6.3.2](../spec/02-trainer.md#632-選定確定と層-1-見送り記録) の「この銘柄で選定」確定時
- 保存先: `Session.candidates[].見送り理由`(自由記述)
- **任意**(素早く選定したいユーザを妨げない)

**層2: エントリー見送り** - 最終候補まで絞ったが「入らない」と判断したケース
- 入力タイミング: [§7.3](../spec/02-trainer.md#73-見送り判断時の追加入力層-2-エントリー見送り) のトレーニング画面「見送る」アクション
- 保存先: `Session.final_decision.エントリー見送り理由`(自由記述)
- **必須**(訓練価値重視)

**全候補見送り** - 銘柄選定画面で 1 つも選ばずにセッション終了
- 入力タイミング: 銘柄選定画面の「全候補を見送り」
- 保存先: `Session.final_decision.skip_reason`(自由記述、任意)

### 9.2 事後データ表示
セッション終了時、見送ったペアについて以下を表示:
- **3段階の事後チャート**(10本先 / 50本先 / 200本先)
- 各期間の最大上昇幅・最大下落幅
- 買い/売り想定時の推定成績
- 実際にエントリーしたペアとの比較

### 9.3 事後評価の算出方針

自己評価タグは持たない([§7.6](../spec/02-trainer.md#76-タグ構造化選択式入力を採用しない方針))。セッション時刻を起点にした事後 OHLC から正解/機会損失を機械的に判定する。

**共通パラメータ(設定で調整可)**

- `lookahead_bars`: 事後評価に使う本数(初期値 50 本)
- `opportunity_pips`: この pips 以上動いたら「機会損失」と見なす閾値(銘柄別、初期値は JPY クロス 20pips / ドルストレート 20pips 相当)
- `noise_pips`: この pips 以下の動きは「どちらでもない」と見なす閾値(初期値 10pips)

**層 1(候補外し)判定**

- `機会損失`: skip 後 `lookahead_bars` 以内に、想定方向に `opportunity_pips` 以上動いた
- `正解`: skip 後 `lookahead_bars` 内の最大変動が `noise_pips` 以下
- `どちらでも`: 上記のどちらにも該当せず(中間帯)
- 想定方向が記録されていない場合は、絶対値ベース(最大上昇・最大下落の大きい方)で判定

**層 2(エントリー見送り)判定**

- `機会損失_エントリー`: 見送り後 `lookahead_bars` 以内に、想定した買い/売り方向に `opportunity_pips` 以上動いた
- `正解_ノーエントリー`: 最大逆行 / 最大順行のどちらも `noise_pips` 以下
- `正解_方向判断`: 想定方向と逆に `opportunity_pips` 以上動いた(入らなくて救われた)
- `慎重すぎ`: 想定方向に `noise_pips` を超えて `opportunity_pips` 未満の動きがあった(微益のチャンスは見えていた)

**層 2 で想定方向を持つ方法**: §7.3 の自由記述「見送り理由」本文から方向性を推定するのは§11 AI 分析の役目。ここでの機械判定には使わず、判定は純粋に OHLC の数値事実で行う。

### 9.4 単一セッションの事後振り返り表示

§9.3 の機械判定は **1 セッション単位の振り返り表示** にのみ使う。横断集計(見送り正解率・機会損失率・慎重すぎ率 等の分母を取って平均する指標)は提供しない([§10 参照](#10-集計機能を採用しない方針))。

セッション終了後の振り返り画面では、そのセッションに限って以下を表示する:
- 見送った候補ごとの判定ラベル(`機会損失` / `正解` / `どちらでも` 等)と最大絶対変動 pips
- 事後 OHLC を用いた 10/50/200 本先チャート(§9.2)
- エントリー済みの場合: `pips_pnl`、SL/TP 到達位置

数値はその場で確認して自分で解釈し、必要なら次のセッションの判断材料にする。複数セッションを横断した勝率・期待値・傾向分析は AI・ローカル共に持たない。

## 10. 集計機能を採用しない方針

本仕様では **複数トレード・複数セッションを横断した集計・統計指標を採用しない**。具体的に不採用とするのは以下:

- 勝率・プロフィットファクター・期待値・最大連勝/連敗などの基本指標
- 後悔指標(最大利益取得率・最大逆行許容度)の集計
- シナリオ別・時間軸別・銘柄別・スタイル別などの多次元集計
- セッション別勝率・曜日別勝率・指標絡み成績
- 見送り正解率・機会損失率・慎重すぎ率などの横断集計(§9.4 の個別表示のみ残す)
- 連続学習日数・週次/月次レポート

**理由:**
- **訓練の目的は 1 判断ごとの質を磨くこと** であり、集計値を追うと「成績を上げる」動機に変質してしまう。勝率や期待値を見ると、そこに最適化する行動が生まれて訓練価値が歪む
- **ノイズに対してサンプル数が不足**(個人の訓練では数百〜千単位のトレードは早々に溜まらず、勝率や期待値はブレが大きく意味を持ちにくい)
- **横断集計が要らないなら、集計アプリ(trade-analyzer)も不要**。分析は §11 の AI 分析(1 セッション単位の観察)で完結させる
- ユーザー自身が必要なら、生データ(セッション履歴・トレード履歴)をエクスポートして外部ツール(Excel 等)で計算できる

**許容される記録機能**(集計ではなく、ユーザーが 1 件ずつ読み返すための履歴):
- セッション履歴一覧(日付・銘柄・判断種別・結果)
- 過去セッションの詳細画面(メモ・描画・事後チャートの再表示)

これらは「集計」ではなく「履歴ブラウジング」の範疇として残す。

## 11. AI分析機能

### 11.1 位置づけ

AI 分析は **1 セッション(= 1 トレード / 1 見送り判断)単位の深掘り観察** のみ提供する。複数トレード・複数セッションを横断した集計・統計・傾向分析は [§10](#10-集計機能を採用しない方針) で不採用と決めており、AI 側でも同様に行わない。

- **発見型のみ**: ユーザーが自覚していない視点の抜け、メモの曖昧さ、シナリオと実展開のズレを観察として提示する
- **コーチングはしない**: 「次はこうすべき」「改善案」は返さない。処方はユーザー自身が考える(訓練価値の保全)
- **集計はしない**: 期間横断の傾向抽出・複数トレード比較・勝率相関は AI には投げない(そもそも §10 で集計機能自体を持たない)

### 11.2 分析対象

Session 単位で実行する。

- **主対象**: `Session.final_decision.symbol` の **判断時点までの** メモ群
  - エントリー時: Scenario の判断時点項目(共通メモ + 3 シナリオ + 波動カウント + エントリー根拠 + スタイル選定理由)+ Trade の判断時点数値(エントリー時刻・価格・SL・TP・方向)
  - 層 2 見送り時: 共通メモ + `エントリー見送り理由` + 検討したスタイル
- **副次情報**: 層 1 候補外し(選定銘柄と外した候補の判断時点以降の値動きを対比させ、選定判断そのものを観察対象に含める)
- **全候補見送り**(§9.1)も対象。共通メモと `skip_reason` を元に「入れる銘柄が無いと判断した根拠」を観察する
- **送らない情報**: 損益(`pips_pnl`)・決済理由・§9.3 の機械判定ラベル・決済時メモ・振り返りメモ([§11.9](#119-訓練価値保全のためのプロンプト設計方針) 参照)

### 11.3 入力データ

#### 11.3.1 チャート画像
- 判断時点(提示日時 / エントリー時 / 見送り時)に **ユーザーが表示していた TF** を各 1 枚ずつレンダリングして送信
- 描画(トレンドライン・水平線・フィボナッチ)は画像に焼き込んだ状態で書き出す
- エントリー / 決済 / 見送り位置のマーカーも画像に焼き込む
- lightweight-charts のスクリーンショット API を利用してフロントエンドで書き出し

#### 11.3.2 最小メタデータ
画像では正確に読めない値を補う最小限のテキスト:
- エントリー / 見送り判断の時刻・価格(**判断時点の情報のみ**)
- SL / TP 価格(判断時点で設定した値)
- スタイル情報(選択スタイル・スタイル選定理由)
- スタイル想定値との対比のための**保有時間・実 SL 幅・実 TP 幅**(エントリー済みで決済済みの場合のみ。pips 損益は含まない)
- 近接経済指標(判断時刻前後 ±N時間、未来分を含む。実戦で事前取得可能な情報の扱い)
- インジケーター設定(種類・期間、§11.8 スナップショット方式)
- 描画の種類と作成 TF(座標は画像に焼き込み済みのため不要)

**送らない**: `pips_pnl`・`exit_reason`・勝敗フラグ・§9.3 の機械判定ラベル(`機会損失`・`慎重すぎ` 等)。これらを送ると AI の観察が結果論に引きずられるため([§11.9](#119-訓練価値保全のためのプロンプト設計方針))。

#### 11.3.3 メモ全文

**送る**:
- 共通メモ(環境認識 / 相場観 / 指標認識 / 銘柄選定理由 / 比較候補の総評)
- エントリー時: 3 シナリオ(メイン + 代替 1 + 代替 2)+ 波動カウント + エントリー根拠 + スタイル選定理由
- 見送り時: 見送り理由 + 検討したスタイル
- 層 1 候補のメモ(各候補の `candidate memo` + 見送り理由)

**送らない**: 決済時メモ・振り返りメモ(判断後・時間経過後に書かれたもの。結果論・事後省察が混じるため AI 観察の対象から外し、ユーザー自身の省察資料として保全する)。

### 11.4 出力形式(視点別セクション + 自問)

Markdown レポートを返す。以下の視点セクションで構成する(該当する観察がない場合は「特記事項なし」と明示する)。視点は **判断時点の情報品質** を中心に置き、結果論観察を避ける。

- **記入シナリオと判断後の動きの対応**: 記入シナリオ(メイン・代替 1・代替 2)と判断後の実チャートの動きの対応関係を事実として記述。どれにも該当しない動きだった場合はそれ自体を観察として返す(勝敗の断定・評価はしない)
- **エントリー根拠と判断時点チャートの整合**: 根拠として挙がったパターン・ライン・指標が判断時点の画像上で実際に成立していたか(時系列的に判断時点で成立していなかったパターンを根拠にしていないか)
- **メモの曖昧表現・視点の欠落**: 「なんとなく」「良さそう」等の曖昧語、時間軸を特定しない表現、代替シナリオ 2 の空欄、相場観が薄い 等
- **スタイル選定と実運用のズレ**(決済済みエントリーのみ): 想定保有時間 / 想定 RR / 典型 SL 幅と実績(保有時間・実 SL / TP 幅)の差を事実として記述。損益の勝敗には踏み込まない
- **層 1 候補外しの対比**(副次): 選定銘柄と外した候補の判断時点以降の値動きを並べて提示。どちらが大きく動いたかは事実として記述するが、「外すべきでなかった」等の断定はしない

末尾に **「自問すべき問い」** を 3〜5 個添える(処方ではなく、ユーザーが自分で考え直すための起点)。

例:
- 「メインシナリオと代替シナリオ 1 は方向が逆だが、入る時点でどちらの根拠がより強いと感じていたか?」
- 「エントリー根拠の『ダブルトップ』は H1 で見ても同じ判断だったか?」
- 「層 1 で外した EURJPY は判断後に USDJPY より大きく動いた。選定時に両者を分けた判断要素は何だったか?」

### 11.5 呼び出しタイミングと対象アプリ

- **オンデマンドのみ**。セッション直後の自動実行は行わない(コスト抑制・ノイズ回避)
- 振り返り画面 / セッション一覧の各行から「AI 分析」ボタンで起動
- **アプリ B (trade-trainer)** と **アプリ C (trade-live)** の両方で利用可能。どちらも Session 単位の分析で、mode(training / real)情報をメタデータに含める
- **集計専用アプリは持たない**([§10](#10-集計機能を採用しない方針))。AI 分析は trade-trainer 本体の振り返り画面から呼び出す

### 11.6 データ送信とコスト・プライバシー管理

**送信前プレビュー(必須ゲート)**
- 分析実行前に「送信予定データ」のプレビュー画面を表示(画像・メタデータ・メモ全文)
- チェックボックスで個別要素(特定メモ・画像の一部)の送信除外が可能
- 「送信」ボタンを押して初めて API 実行
- 一度送信したデータは取り戻せないため、実行前確認を必須化

**バジェット管理**
- 設定画面で月額バジェット上限を指定(デフォルト 2000 円相当)
- 実行前に推定トークン数・推定コストを表示
- 上限到達時は実行不可(翌月リセット)
- 利用量ダッシュボード: 当月 / 先月の実行回数・送信トークン数・推定コストを可視化

**キャッシュ**
- 入力データ(画像バイト列を含む)のハッシュでキャッシュ
- 同一入力の再分析はキャッシュから返す(API コールしない)
- ユーザーが明示的に「再実行」を選んだ場合のみ再コール

**時刻情報の扱い**
- 送信データ内の日時は ISO8601(UTC)で送信
- システムプロンプトで「ユーザーは JST で運用しており、メモ本文内の時刻は JST 基準」と明記

### 11.7 分析結果の永続化(ファイルストレージ)

AI 分析結果は DB ではなく **ファイルストレージ** に保存する。統計・集計対象ではなく、検索パターンが「Session から紐付く結果を開く」のみのため、リレーショナル検索の利点が発揮されず、画像・長文 Markdown を blob カラムに置くと DB が肥大化するため。

**レイアウト**

```
data/ai_analysis/
  {session_id}/
    index.json                    # 履歴リスト [{ timestamp, hash, tokens, cost, model }]
    {timestamp}_{short_hash}/
      input.json                  # 送信メタデータ + メモ本文スナップショット
      images/{tf}.png             # 送信画像(TF ごと、描画焼き込み済み)
      report.md                   # AI が返した Markdown レポート
      meta.json                   # { hash, tokens, cost, model, created_at }
```

**運用**

- **再表示**: `{session_id}/index.json` から最新エントリを引き、該当ディレクトリの `report.md` を表示(API 再コールしない)
- **キャッシュ判定**: 新規実行時は送信予定 payload(メタ + メモ + 画像バイト)のハッシュを計算し、`index.json` に同一 hash があればそのディレクトリを返す
- **再実行**: ユーザー明示の「再実行」時は新ディレクトリを追加(旧結果はそのまま残し、内容変化を diff で比較できる)
- **差分比較 UI**: `index.json` の時系列から任意の 2 エントリを選び、`report.md` を並列表示

**DB との接点**

- Session との紐付けはファイルパス(`{session_id}/`)のみで表現し、DB テーブルは持たない
- セッション削除時はファイル群も連動削除する(アプリ側の責務)
- バックアップは DB と別経路(ファイルコピー / アーカイブ)で行える

### 11.8 分析コンテキストの導出方針

AI 送信時に必要な市場コンテキストは、**DBへの冗長保存は行わず、判断時刻(`presented_at` / `entry_time` / `exit_time`)を起点に on-demand で導出**する方針を貫く。

| 導出項目 | 導出元 | 実装場所 |
|---|---|---|
| チャート画像(各表示 TF、描画・マーカー焼き込み済み) | OHLC + `Drawing` + `IndicatorConfig` を lightweight-charts でレンダリング | フロントエンド送信時 |
| エントリー / 決済 / 見送り時点の数値メタデータ | `ohlc_m5` + `market-data` のリサンプル | `market-data.accessor.get_ohlc()` |
| MAE / MFE(最大含み損・含み益) | entry_time〜exit_time の M5 OHLC から min / max 計算 | `market-data` ヘルパー |
| 事後 OHLC 要点(最大順行・逆行 pips、§9.3 機械判定) | 判断時刻〜 lookahead 期間の M5 OHLC | `market-data` ヘルパー |
| 近接する経済指標 | `economic_event` を判断時刻前後 ±N時間で検索 | クエリ関数 |
| セッション時間帯(東京 / ロンドン / NY) | 判断時刻の UTC から判定 | ユーティリティ関数 |
| インジケーター値(SMA / EMA / RSI 等) | OHLC + `IndicatorConfig` から再計算 | 下記スナップショット参照 |

**例外:インジケーター設定スナップショット**

ユーザーが後から期間・種類設定を変更すると、過去トレード分析時に当時の値を再現できない。対策として:

- `Setting` にインジケーター設定リストを保持し、変更時に **新バージョンを追記(履歴保持)** する
- `TradeSession` に `indicator_config_id`(その時点の設定バージョン ID)を紐付け
- 分析時はこの ID を介して当時の設定を取得し、OHLC から値を再計算

この方式により、描画オブジェクト・メモ等の **ユーザー入力のみを保存** し、市場コンテキスト(画像含む)は都度導出・レンダリングで再現する設計を貫徹する。

### 11.9 訓練価値保全のためのプロンプト設計方針

AI 分析が訓練価値を毀損しないよう、プロンプト段階で以下のガードレールを敷く。仕様書レベルで明文化し、実装時のシステムプロンプトに展開する。

#### 11.9.1 AI に与える役割定義

- ユーザーは判断の質を自分で磨きたいトレーナー
- AI は **観察者** として、事実・対応・整合を提示するのみ
- 改善策の提案・勝敗の評価・価格予測・メモの代弁は役割の外
- ユーザーが自分で気づき、自分で処方を考えるための **観察材料を提供する** ことが役割

#### 11.9.2 禁止する表現・行為

- **処方・評価**: 「〜すべき」「〜が良い / 悪い」「正解は〜」「間違いは〜」「次は〜」「改善策」「推奨は〜」
- **価格予測**: 「この後〜方向に動くはず」「〜円まで伸びる可能性」
- **メモの代弁**: 「あなたは〜と考えていた」「本心では〜と感じていたはず」(メモの事実記述で留める)
- **結果論観察**: 画像から勝敗が推測できても、勝敗を踏まえた観察はしない(例: 「結果的に負けたのでこの根拠は弱かった」は禁止)
- **判定ラベルの引用**: 送信データに含まれない `機会損失`・`慎重すぎ` 等の §9.3 ラベルを AI が再発明して断定するのも禁止(値動きの事実のみ記述)
- **想定を超えた断定**: 「この場面は典型的な〜」「一般的には〜」等、観察対象を離れた一般論

#### 11.9.3 推奨する表現・形式

- 事実記述: 「メモには X と書かれている」「画像では Y の形が観察される」「両者の対応は Z」「X は空欄」
- 対応関係: 「シナリオ A と実動きは方向が一致しているが、到達水準は異なる」
- 差分提示: 「想定保有時間 1 時間に対し、実績は 5 時間」
- 空欄・欠落指摘: 「代替シナリオ 2 が空欄」「相場観の記述が 1 文のみ」

#### 11.9.4 自問セクションの生成方針

- 処方ではなく **省察の起点** として機能する問いにする
- 「〜すべきか」「〜した方が良いか」ではなく、「どう感じていたか」「見ていたか」「どちらが強いと感じたか」の形に統一
- AI の質問にユーザーが答える必要はない(省察資料として読み、自分の中で再考する)

#### 11.9.5 送信データによるガード(多層防御)

プロンプトの規範だけでなく、送信データ側でも結果論観察を物理的に抑止する([§11.3](#113-入力データ) 参照):

- 損益・勝敗・決済理由・判定ラベルはそもそも送らない
- 決済時メモ・振り返りメモは送らない(判断後・事後省察を AI の観察対象から除外)
- 画像から勝敗が推測できる点は避けられないが、プロンプト方針 §11.9.2 で「勝敗に踏み込まない」を明示することで観察の軸を判断時点の情報品質に固定する

# FXトレーニングアプリ 仕様書

## 1. 概要

### 1.1 目的
チャート分析・エントリー判断・エリオット波動分析のスキルを実戦的に訓練するための個人用アプリ。

### 1.2 コンセプト
- **時間起点の銘柄選定**: 特定の日時を先に提示し、その時点で優位性のある銘柄をユーザー自身が選ぶ
- **トレードスタイル選択**: 短期・中期・指標狙いなど複数のスタイルを状況に応じて使い分ける訓練
- **シナリオメモ主導**: 思考プロセスを必ず言語化し、後から振り返り可能にする
- **見送りも評価**: 「入らなかった判断」も記録・評価の対象
- **経済指標の認識訓練**: 強指標をチャート上に可視化、相場の背景要因を意識した判断を促す
- **トレーニングとリアルの一貫性**: 同じ分析・メモ機能を実トレードでも使用可能
- **AI分析支援**: 蓄積データから傾向・改善点を自動抽出

### 1.3 対象ユーザー
自分専用(単一ユーザー)

### 1.4 プラットフォーム
Web + スマホ両対応(レスポンシブデザイン)

### 1.5 システム構成(コンポーネント分割)

本システムは**3つの独立アプリ + 3つの共通パッケージ**で構成し、全てを**Windows VPS(XServer VPS等)一台で稼働**させる。モノリシックアプリではなく責務ごとに分割することで、開発の段階化・リスクの局所化・再利用性を確保する。

```
[クライアント]
  スマホ / PC のWebブラウザ
     ↓ HTTPS
[Windows VPS (24/365稼働)]
  ├─ MT5本体 (常時起動、RDPで管理)
  │
  ├─ [共通パッケージ]
  │    shared-schema        ─ DB接続・スキーマ定義・マイグレーション
  │    common-ui-lib        ─ チャート/メモフォーム/描画ツール等の共有UI
  │    market-data          ─ 市場データ取得ライブラリ(ハイブリッドキャッシュ)
  │
  ├─ [共有ストア]
  │    SQLite (trading.db)  ─ 全アプリが参照する単一ファイル
  │
  ├─ [アプリ B] trade-trainer (Webサーバー)
  │    └ 過去チャートを使った訓練。発注コードを一切含まない
  │
  ├─ [アプリ C] trade-journal (Webサーバー)
  │    └ 実トレードの実行。MT5経由で発注、ポジション管理
  │
  └─ [アプリ D] trade-analyzer (Webサーバー)
       └ 蓄積データの集計・AI分析・レポート生成
     
     ↓ MT5 Python API (同一マシン内、遅延数ms)
  [ブローカーサーバー]
```

**市場データ取得の方針: ハイブリッドキャッシュ**

価格データは**事前バルク取得せず**、使われた範囲を自動的にDBキャッシュする遅延ロード方式を採る。

```
[アプリが価格データを要求]
     ↓
[market-data ライブラリ]
  ├─ SQLiteキャッシュを確認
  │   ├─ ある → キャッシュから返す(高速)
  │   └─ ない → MT5から取得 → SQLiteに保存 → 返す
  ↓
[アプリ]
```

- **初回セットアップ不要**: 使いながら自動的にキャッシュが溜まる
- **MT5依存は初回のみ**: 一度使った範囲はMT5が落ちていても利用可能
- **経済指標データは事前取得**: 価格と異なり量が少なく、日次バッチ更新で済む(market-dataのCLIで実行)

**分割の基本方針**
- **全アプリがVPS内で稼働**: MT5・アプリ・DBが同居、発注遅延がほぼゼロ
- **発注コードの隔離**: 誤発注防止のため B には発注コードを一切含めない。発注機能は C のみ
- **UIの共通化**: チャート・メモ・描画ツール等は `common-ui-lib` として B/C/D で共有
- **データ取得の共通化**: 市場データ取得は `market-data` ライブラリに集約、B/C から呼び出し
- **データ単一化**: 3アプリ全てが同じSQLiteファイルを参照。スキーマは `shared-schema` で統一管理
- **モバイル完全対応**: クライアントはブラウザのみ、スマホからHTTPS経由で全機能利用可能

**各コンポーネントの関係**

| コンポーネント | 種別 | 実行環境 | DB操作 | MT5接続 | 発注機能 | 開発Phase |
|---|---|---|---|---|---|---|
| shared-schema | パッケージ | - | - | - | - | Phase 1 |
| common-ui-lib | パッケージ | - | - | - | - | Phase 2 |
| market-data | パッケージ | - | 読み書き(市場データキャッシュ) | 取得 | - | Phase 1 |
| B. trade-trainer | Webアプリ | VPS | 読み書き(トレーニング記録) | 経由(market-data 経由) | **なし** | Phase 1〜3 |
| C. trade-journal | Webアプリ | VPS | 読み書き(リアル記録) | 直接(発注) + 経由(データ) | あり | Phase 5〜6 |
| D. trade-analyzer | Webアプリ | VPS | 読み込み(全データ) | なし | なし | Phase 4 |

**開発環境と本番環境**
- 本番: Windows VPS
- 開発: 手元PCでもアプリB/Dは動作可能(MT5非依存、market-dataはキャッシュだけ参照するモードで起動)。アプリC および market-data の MT5取得パスはVPS上で開発・テスト

**DBスキーマ管理**
`shared-schema` パッケージに以下を持たせる:
- SQLAlchemy によるスキーマ定義
- alembic マイグレーションスクリプト
- 3アプリ + market-data がこのパッケージを依存関係として使う

スキーマ変更時は `shared-schema` を更新し、全アプリをそれに追従させる。

## 2. データ基盤

### 2.1 データソース
- **初期実装**: MT5 (`MetaTrader5` Python パッケージ)
- **将来拡張候補**(MT5で必要データが取得できない場合): Dukascopy, HistData, TrueFX 等
- **market-data** パッケージが**データソース抽象化**されており、設定ファイルで切り替え可能

### 2.2 保存データ(市場データ)
- **ハイブリッドキャッシュ方式**: 事前バルク取得は行わず、使われた範囲を自動的にSQLiteにキャッシュ
- **M5(5分足)を最小単位として管理**
- ユーザーのトレードスタイルの最小時間軸がM5のため、M1は保存しない
- 上位足(M15/M30/H1/H4/D1等)は動的集約で生成
- **最小保存足は設定可能**: 将来スキャルピング訓練等でM1が必要になった場合は設定変更で対応
- **DB**: SQLite(WALモードで読み書きの並行性が良い。3アプリでの共有に適する)
- **データの扱い**: キャッシュレコードには取得日時・データソースを記録、後から再取得判定が可能

### 2.3 保存データ規模の見積もり
- 使われた範囲のみキャッシュする方式のため、初期は小さく徐々に増える
- 全期間キャッシュが蓄積された場合の上限: M5で5年分1銘柄あたり約40〜60 MB
- 10銘柄でも最大400〜600 MB、30銘柄でも2 GB未満
- SQLiteの上限(理論上 281 TB)から見て余裕、実用上問題なし

### 2.4 データソース抽象化の設計
全てのデータソースは共通インターフェースを実装し、差し替え可能とする。

```python
class DataSourceProvider:
    def fetch_ohlc(symbol, timeframe, from_dt, to_dt) -> DataFrame: ...
    def fetch_latest(symbol, timeframe) -> DataFrame: ...
    def get_available_range(symbol) -> (from_dt, to_dt): ...
```

**正規化ルール**(プロバイダー内部で吸収)
- タイムゾーン: **データ保存は全てUTC**に統一(表示は後述のUI層でJSTに変換)
- 価格基準: 全て**Bid基準**に統一
- 銘柄名: 接尾辞を除去した統一形式(例: `USDJPY.a` → `USDJPY`)
- 出来高: ティック数または出来高(プロバイダー差異はメタ情報で管理)

### 2.5 DBスキーマでのソース管理
複数ソース共存を可能にするため、`source` カラムを設ける。

```sql
CREATE TABLE ohlc_m5 (
  symbol TEXT,
  timestamp TIMESTAMP,
  open DOUBLE, high DOUBLE, low DOUBLE, close DOUBLE,
  volume BIGINT,
  source TEXT,  -- 'mt5', 'dukascopy', etc.
  PRIMARY KEY (symbol, timestamp, source)
);
```

これにより、途中でソースを切り替えても過去データを活かせる。

### 2.6 トレード記録との整合性
データソース変更時に既存のトレード記録が影響を受けないよう、以下を守る:
- エントリー時点の価格は**スナップショットとしてトレード記録に保存**
- スプレッドは**銘柄別固定値**を使用(ソース依存しない)

### 2.7 取得対象期間
- **想定レンジ**: 過去5年程度までを訓練対象とする
- 事前バルク取得はしないため、ユーザーがトレーニング中に選んだ期間だけが段階的にキャッシュされる
- MT5が保持しているデータ範囲はブローカー依存のため、Phase 1 で主要銘柄の過去5年分が取得可能かを**検証**する
- 取得困難な期間がある場合は Dukascopy 等への切り替えを検討

### 2.8 対象銘柄
- 設定画面でユーザーが指定
- 有効化した銘柄群を出題対象として使用

### 2.9 データ更新
- **価格データ**: ユーザーのアクセス時に market-data ライブラリが自動取得・キャッシュ(明示的な更新ボタンは不要)
- **経済指標データ**: Windowsタスクスケジューラで market-data CLI を日次実行、新しい指標予定を取り込む
- **手動更新**: アプリUIから market-data CLI を呼び出す機能も提供(設定画面等)

### 2.10 経済指標データ
- **第一候補**: MT5 Economic Calendar API (`calendar_value_history()` 等)
- **代替**: 外部CSVの事前取り込み(ブローカーのMT5環境で取れない場合)
- **取得方式**: 事前バッチ取得(market-data CLIの日次実行)。価格と違いデータ量が少なく、予定値を含めて事前に知りたいため
- **保存内容**: 発表日時、通貨、指標名、重要度、実測値、予想値、前回値
- **重要度扱い**: MT5の `Importance` フィールドを使用(3段階)
- **時刻精度の注意**: 最小保存足がM5のため、指標発表時刻を含むM5足が1単位となる。発表前後の細かい1〜2分単位の動きは確認できないが、ボラの塊としては十分可視化される

### 2.11 タイムゾーン表示ポリシー
- **保存層**: 全データを**UTC**で保存・処理(DB・API・バックエンド)
- **表示層**: ユーザー向け表示は全て**JST(Asia/Tokyo)**に変換
- **変換責務**: フロントエンド側で変換(`dayjs` + `dayjs/plugin/timezone` または `date-fns-tz` 等)
- **変換対象**:
  - チャート時刻軸・クロスヘア・ツールチップ
  - 経済指標の発表時刻表示
  - セッション日時・トレード記録の日時表示
  - シナリオメモの入力時刻・保存時刻
  - CSVエクスポートの日時(JST表記で出力)
- **入力層**: 時間フィルタ等の時刻入力もJSTで解釈(入力UIに「(JST)」と明記)
- **シナリオメモ本文中の時刻記述**: ユーザーは時刻を**JSTで記述**する前提。例「23時のFOMC前に利確」「翌朝9時に再評価」。システムは本文内の時刻を自動変換しない。入力UIに「時刻はJST」とプレースホルダー表示。AI分析にメモを渡す際もプロンプトに「メモ内時刻はJST基準」と明記する
- **セッションプリセット**(JST基準、夏時間は考慮せず固定):
  - 東京: 09:00〜15:00
  - ロンドン: 16:00〜25:00
  - NY: 22:00〜翌06:00

## 4. セッションフロー

### 4.1 通常セッション(標準フロー)

```
[Phase 1: 時間選定]
1. 時間フィルタ設定(セッション・曜日・期間など)
2. システムがランダムな日時を提示

[Phase 2: 銘柄選定]
3. 対象ペア一覧画面(サムネイルチャート表示)
4. 複数候補をウォッチリストに追加
5. 候補を詳細チャートで比較
6. 1つに絞る(または全候補を見送り)

[Phase 3: 分析]
7. 選んだペアのマルチタイムフレームチャート表示
8. 描画ツール・インジケーターで分析

[Phase 4: シナリオ入力]
9. シナリオメモ入力(必須項目あり)
10. 3シナリオ事前宣言

[Phase 5: エントリー判断]
11. 買い/売り/見送り選択
12. SL/TP設定(エントリー時)

[Phase 6: 進行・決済]
13. ローソク足を進めて裁量決済
14. 決済時メモ入力

[Phase 7: 振り返り]
15. 決済後の続き観察(追加50本)
16. 後悔指標表示
17. 見送り事後検証
18. 自己評価タグ付け
19. 総合振り返りメモ
```

### 4.2 反復モード
過去に判断した同じチャートを再出題。判断のブレや手法の再現性を検証する。

### 4.3 セッション管理
- 時間制限なし
- 途中保存・再開機能あり
- 1セッション1エントリー完結

## 5. チャート機能

### 5.1 表示
- **マルチタイムフレーム同時表示**(メイン時間軸 + 上位足2〜3枚)
- 未来部分は隠蔽、ローソク足を1本ずつ進行可能
- 上位足は形成途中の足をリアルタイム更新(メイン足の進行に同期)
- 時間軸セットはプリセット可能(例: M15メイン + H1/H4/D1)
- **時刻表示はJST**: 時間軸ラベル、クロスヘア、ツールチップ全てJST表示(内部データはUTC保持、表示時のみ変換)

### 5.2 インジケーター(自前実装)
- SMA(単純移動平均)
- EMA(指数移動平均)
- RSI(サブパネル表示)
- 拡張可能な設計で実装

### 5.3 描画ツール
| ツール | 実装 |
|---|---|
| 水平線 | ライブラリ標準 |
| トレンドライン | ライブラリ標準 |
| フィボナッチリトレースメント | プラグイン自前実装 |
| テキストラベル(波動カウント用) | プラグイン自前実装、連続クリックで自動採番支援 |

### 5.4 経済指標表示
- **表示対象**: 重要度 **星3のみ**(設定で星2以上への拡張も可能な設計)
- **通貨フィルタ**: 表示中チャートの通貨ペアに関連する通貨 + USD(基軸通貨)
- **表示スタイル**:
  - **縦線マーカー**: 発表時刻に縦線(通貨別に色分け、ホバーで詳細表示)
  - **シェーディング**: 発表前後の時間帯(例: 前5分〜後30分)を背景色で塗り、ボラ上昇期間を可視化
- **時間軸上の情報公開ポリシー**(実戦で得られる情報と同じ扱い):
  - **未来の指標も表示**: 発表予定時刻、指標名、通貨、重要度、**予想値**、前回値はセッション現在時刻より未来でも表示(実戦で事前に知れる情報)
  - **実測値のみ発表時刻到達後に表示**: サプライズ度も実測値公開後に算出・表示
  - 縦線・シェーディングは未来分も含めて常時表示(次の指標までの残り時間を意識できる)
- **ホバー情報**: 指標名、通貨、発表時刻、実測値/予想値/前回値、サプライズ度(実測 vs 予想の乖離)

### 5.5 描画の保存
セッションごとに描画オブジェクトを保存し、後から視覚的に振り返り可能。

### 5.6 技術選定
- **TradingView Lightweight Charts** + プラグイン自前実装
- React でラップ

## 6. 銘柄選定画面

### 6.1 サムネイル一覧
- 対象ペア全ての**ミニチャート**をグリッド表示
- 各ペアに複数時間軸(例: H4 + D1)のミニチャートを並列
- 直近変動率を数値表示
- 時間軸切り替え可能

### 6.2 候補管理
- ペアクリックで「候補に追加」
- 複数候補を**ウォッチリスト化**
- 候補同士を詳細チャートで比較
- 最終的に1つ選定 or 全候補見送り

### 6.3 通貨強弱について
**自動算出・ダッシュボードは実装しない**。ユーザーが生チャートから自分の目で強弱を判断する訓練方針。メモに自分の読みを記録する形で記録。

## 8. トレードスタイル管理

状況に応じて複数のトレードスタイルを使い分ける訓練のため、スタイルを構造化して管理する。

### 8.1 スタイルの考え方
- 短期・中期・指標狙いなど、トレードの性格が異なる複数のスタイルを定義
- セッション開始時ではなく**分析後・エントリー前に選択**(状況判断の訓練として)
- スタイル選択もシナリオメモに記録し、後から集計・分析

### 8.2 スタイル定義(ユーザー定義可能)
ユーザーが設定画面から追加・編集・無効化可能。

**スタイル定義項目**

| 項目 | 内容 | 例 |
|---|---|---|
| id | 識別子 | `short`, `mid`, `news` |
| name | 表示名 | 短期トレード、中期トレード、指標トレード |
| primary_timeframe | 主要時間軸 | M5, H1 |
| expected_hold_time | 想定保有時間 | 10分〜1時間、数時間〜1日 |
| expected_rr | 想定リスクリワード | 1:1.5, 1:2 |
| typical_sl_pips | 典型的なSL幅 | 10〜20pips, 30〜50pips |
| description | 運用ルール・説明 | 自由記述 |
| is_active | 有効/無効フラグ | true/false |

### 8.3 初期プリセット(例)
新規ユーザー向けに以下をプリセットとして提供。ユーザーが自由に編集・削除可能。

- **短期トレード** (`short`): M5基準、10分〜1時間保有、RR1:1.5、SL10〜20pips
- **中期トレード** (`mid`): H1基準、数時間〜1日保有、RR1:2、SL30〜50pips
- **指標トレード** (`news`): 強指標発表前後、M5/M15基準、特殊戦略
- **スイング** (`swing`): H4/D1基準、数日保有(任意)

### 8.4 スタイル選択UI
- エントリー前のシナリオメモ入力画面で**スタイルをドロップダウン選択**
- 選択したスタイルの想定SL幅・RRが自動表示され、SL/TP設定の参考になる
- ただし自動入力はせず、ユーザー自身がSL/TPを入力(想定と実際のズレを計測するため)

### 8.5 見送り時のスタイル記録
見送りを選んだ場合でも、**検討したスタイル**を記録できる。
- 「短期で入ろうとしたが条件合わず見送り」
- 「中期視点では入れるが短期ではダメ」等の判断を残す

## 9. 見送り検証機能

### 9.1 2層の見送り記録

**層1: 候補外し** - ウォッチリストに入れたが最終選定しなかったペア
**層2: エントリー見送り** - 最終候補まで絞ったが「入らない」と判断したケース

### 9.2 事後データ表示
セッション終了時、見送ったペアについて以下を表示:
- **3段階の事後チャート**(10本先 / 50本先 / 200本先)
- 各期間の最大上昇幅・最大下落幅
- 買い/売り想定時の推定成績
- 実際にエントリーしたペアとの比較

### 9.3 自己評価タグ(ユーザー入力)

**層2用タグ(必須)**
- `正解_ノーエントリー`: 入らなくて正解
- `機会損失_エントリー`: 入るべきだった
- `正解_方向判断`: 見送り理由は妥当
- `慎重すぎ`: 過度に慎重だった

**層1用タグ(任意・スキップ可)**
- `正解`: 選ばなくて良かった
- `機会損失`: 選ぶべきだった
- `判断通り`: 選定基準的に外して正しい
- `どちらでも`: 差はなかった

### 9.4 集計指標

**見送り単体指標**
- 見送り正解率
- 機会損失率
- 機会損失の平均逸失pips
- 慎重すぎ指数

**組み合わせ指標**
- 選定精度(エントリー vs 見送り、動きが大きかったのはどちらか)
- 逆選定率
- **総合トレード判断スコア** = (エントリー成功 + 見送り正解) ÷ 総判断数

## 10. 成績評価指標

- 銘柄別成績

- シナリオ的中率(3つのうちどれかに当たった率)
- 選定候補数別勝率

### 10.4 学習継続指標
- 連続学習日数
- 総セッション数
- 週次・月次レポート(自動生成)

- スランプ検出

## 12. リアルトレードアプリ (trade-journal)

トレーニングで鍛えたプロセスをそのまま実戦で使うための**独立アプリ**。トレーニングアプリ(trade-trainer)とは物理的に分離し、UIは `common-ui-lib` を通じて共有する。分析・シナリオメモなど大半の機能はトレーニングと同じ体験になるよう設計するが、発注・ポジション管理・心理状態記録などのリアル特有機能を備える。

### 12.1 アプリ分離の方針
- トレーニングアプリには**発注コードを一切含めない**(誤発注の物理的排除)
- UI・描画ツール・シナリオメモフォーム等は `common-ui-lib` として共有
- DBはトレーニング/リアル共通の `trading.db` を使用。レコードは `mode` フラグで区別
- 視覚的に**リアルアプリの配色を警告色系**にし、トレーニングと一目で区別できるようにする

### 12.2 発注方式
- **完全にMT5経由で発注**(`order_send()` 使用)
- trade-journal が発注エンジンを兼ねる
- 発注結果・約定情報はMT5から取得してDB保存
- **VPS内でMT5とアプリが同居**するため、発注API呼び出しは数ミリ秒で完了
- モバイル(スマホブラウザ)からもVPSへHTTPSでアクセスし、画面で発注指示 → VPS内のMT5で即座に実行
- キュー方式や外部ブリッジは不要(同一マシン内で完結)

### 12.3 リアルトレードアプリのフロー

```
[事前チェック]
1. MT5接続確認、口座残高・証拠金・ポジション状況取得
2. 許容損失・当日ルールの確認

[通常分析(トレーニングと共通のUI/フロー)]
3. 銘柄選定、マルチタイムフレーム分析
4. 経済指標確認
5. シナリオメモ入力(3シナリオ事前宣言、タグ)
6. 心理状態記録

[発注前計算]
7. ロット自動計算
   - 口座残高 × リスク% ÷ (SL幅 × pip価値)
   - 計算結果を確認・微調整可能

[発注]
8. 発注内容の確認ダイアログ(銘柄/方向/ロット/SL/TP/想定損失額)
9. 「確定」ボタンで二段階確認後、MT5 order_send() 実行
10. 約定確認、エラー時はリトライ/中断

[保有中]
11. リアルタイム損益・ポジション状態表示
12. 中間メモ(心理状態変化、シナリオ再評価)
13. シナリオ崩壊アラート(撤退ラインへの接近通知)

[決済]
14. 手動決済 or TP/SL自動約定
15. 決済時メモ(心理状態含む)

[事後]
16. 後悔指標(決済後の動きを一定時間追跡表示)
17. 振り返りメモ
```

### 12.4 リスク管理機能

- **口座設定**: 残高、口座通貨、レバレッジ(MT5から自動取得)
- **リスク設定**: 1トレードあたりの許容損失(%または金額、ユーザー設定)
- **ロット自動計算**: SL幅から推奨ロット算出、ユーザー確認後に確定
- **表示項目**: 発注前に想定損失額を金額で明示

### 12.5 発注安全機構(最小限)

- **二段階確認ダイアログ**: 発注ボタン → 確認画面 → 「確定」
- **シナリオメモ必須**: メモが未入力なら発注不可(衝動取引防止)

※その他の安全機構(異常値検出、スプレッド異常検知、クールダウン、連敗ストッパーなど)は**将来拡張の候補**として保留。必要に応じて後から追加。

### 12.6 心理状態の記録

投資判断の経緯を残すため、心理状態を構造化して記録する。

**エントリー時(必須)**
- 感情: 落ち着き / 焦り / 興奮 / 恐怖 / 退屈 / その他(選択式 + 自由記述)
- 直近成績の影響: 連勝中 / 連敗後 / 平常 / 意識せず
- 身体状態: 良好 / 疲労 / 睡眠不足 / 空腹 / その他

**保有中(任意、タイムラインで複数記録可)**
- 任意のタイミングで心理状態の変化をメモ
- 「想定と違う動きで不安」「含み益で利確したい」など

**決済時(必須)**
- 決済時の感情
- 冷静な判断だったか、感情に押されたか(自己評価)

### 12.7 ポジション継続管理

- ダッシュボードに**オープンポジション一覧**
- 各ポジションに以下を即座に参照可能
  - 当初のシナリオメモ(エントリー根拠、撤退ライン)
  - 心理状態の時系列メモ
  - 現在損益、経過時間
- **シナリオ崩壊アラート**: メモ記載の撤退ラインに近づいたら通知
- 保有中の中間メモ記録UIを常時利用可能

### 12.8 トレーニングとの連携

**trade-journal → trade-trainer**
- 負けたトレード(または任意のトレード)を「この局面を復習」ボタンで trade-trainer に引き渡し、**同じ局面のトレーニングセッション**として再現
- リアルで遭遇したパターンを反復練習に回す
- 引き渡しはDB経由(復習キューテーブルに登録、trainer起動時に表示)

**trade-trainer → trade-journal**
- 共有DBを通じて、トレーニングで高勝率のタグ・パターンを「得意」として参照
- エントリー時、現在のシナリオタグがトレーニング上で低勝率なら警告表示

### 12.9 リアル特有の集計指標

- 実金額ベースのP&L(月次・累計)
- **理論勝率とのギャップ**: リアル勝率 vs 同タグのトレーニング勝率(心理影響の定量化)
- 心理タグ別成績(「焦り」エントリーの勝率など)
- 時間帯別の心理状態と成績の相関
- ポジション保有時間と成績の相関

### 12.10 データ構造
- トレード記録は**トレーニングと共通スキーマ**(`mode: training | real` フラグで区別)
- 集計・AI分析時はモード別に分離可能
- AI分析は原則モード別に評価(性質が違うため)
- ただし「得意パターン抽出」などはモード横断で分析可能

### 12.11 段階的ロールアウト

実トレードはリスクが大きいため、以下の順で慎重に展開する。

1. **MT5デモ口座で検証**: 数ヶ月運用して誤発注・不具合がないことを確認
2. **実口座・最小ロット運用**: 1ヶ月程度、最小ロット固定で安定性確認
3. **通常運用開始**: リスク設定に基づく自動ロット計算を有効化

**Python環境管理: uv**

### 15.2 共通基盤
- **SQLite**(単一ファイルDB、全アプリで共有)
  - **WALモード**で運用(読み書きの並行性確保、マルチアプリ構成に必須)
  - インデックスは `(symbol, timestamp)` を基本に、用途に応じて追加
  - バックアップは単純なファイルコピー(WALモードでは `.backup` コマンド推奨)
  - 選定理由: OLTP向けで本アプリのワークロード(小さな書き込み頻繁、数万行規模の読み込み中心)に適合。枯れたツール群(GUI・ORM・マイグレーション)が揃う
- **shared-schema** パッケージ: SQLAlchemy + alembic でスキーマ定義・マイグレーション管理
- **market-data** パッケージ: 市場データ取得ライブラリ
  - ハイブリッドキャッシュ方式(オンデマンド取得 + SQLite自動キャッシュ)
  - データソース抽象化(MT5、将来Dukascopy等)
  - 経済指標の事前取得(CLI、タスクスケジューラから日次実行)
  - アプリB/Cから呼ばれる
- **common-ui-lib** パッケージ: チャート・描画ツール・シナリオメモフォーム等の共有UIコンポーネント

### 15.3 アプリB: trade-trainer
- バックエンド: **Python** + **FastAPI**
- フロントエンド: **React** + **TypeScript** + **TradingView Lightweight Charts**
- `common-ui-lib` と `market-data`(参照のみ) を使用
- 実行環境: Windows VPS(本番)、手元PC(開発、market-dataはキャッシュ参照モード)
- **発注関連コードを含まない**
- 仮想環境: `apps/trade-trainer/backend/.venv`(uv管理)

### 15.4 アプリC: trade-journal
- バックエンド: **Python** + **FastAPI** + **MetaTrader5** パッケージ
- フロントエンド: **React** + **TypeScript** + `common-ui-lib`
- `market-data` でチャート取得、MT5 Python APIで直接発注
- 実行環境: Windows VPS(MT5と同居、発注遅延ほぼゼロ)
- 配色テーマを警告色系にしてトレーニングと区別
- 仮想環境: `apps/trade-journal/backend/.venv`(uv管理)

### 15.5 アプリD: trade-analyzer
- バックエンド: **Python** + **FastAPI**
- フロントエンド: **React** + **TypeScript** + グラフ描画ライブラリ(Recharts 等)
- AI分析: ローカルPython(定量集計) + **Claude API**(定性分析)
- DB読み込みのみ(MT5接続なし)
- 実行環境: Windows VPS(本番)、手元PC(開発)
- 仮想環境: `apps/trade-analyzer/backend/.venv`(uv管理)

### 15.6 運用構成

**実行環境: Windows VPS(XServer VPS Windows等)一台に集約**

| コンポーネント | 配置 |
|---|---|
| MT5本体 | VPS(常時起動、RDPで初期セットアップ) |
| market-data CLI (経済指標日次更新) | VPS(Windowsタスクスケジューラで日次実行) |
| アプリB (trade-trainer) | VPS(Webサーバーとして常時稼働) |
| アプリC (trade-journal) | VPS(Webサーバーとして常時稼働) |
| アプリD (trade-analyzer) | VPS(Webサーバーとして常時稼働) |
| SQLite DB | VPS(全アプリが同一ファイルを参照) |

**アクセス方法**
- クライアント(スマホ・PC)からHTTPSでVPSのWebサーバーへアクセス
- 認証はVPS側で完結(JWT or セッション)
- モバイル・PC・タブレットどこからでも同じ体験

**VPSスペック目安**
- 推奨メモリ: 4GB以上(MT5 + 3アプリ + DB の同居を考慮)
- ストレージ: 20GB以上(データ蓄積の余裕を見て)
- OS: Windows Server(MT5 Python APIの動作環境)

**MT5の常時稼働**
- Windowsセッション切断後もMT5を稼働させる設定
- MT5が落ちた場合の自動復旧(タスクスケジューラによる監視)
- 定期的な接続状態チェック

**バックアップ**
- SQLite DBファイルを定期的に手元PCへダウンロード
- VPSスナップショット機能(XServer VPSで提供あり)を活用

**開発環境**
- 本番: Windows VPS
- 開発(アプリB/D): 手元PC(Mac/Linux/Windows問わず)で動作可能(market-dataはキャッシュ参照モードで起動)
- 開発(アプリC および market-data のMT5取得パス): VPSにRDP接続して直接開発、またはVPS上のGitリポジトリで作業

## 17. データモデル(概要)

以下のスキーマは `shared-schema` パッケージで定義し、4つのアプリ(A/B/C/D)すべてが共有する。スキーマ変更時はパッケージを更新し、各アプリをそれに追従させる。

```
Session
├─ id, 開始日時, 提示日時(チャート内), 時間フィルタ設定
├─ mode (training | real)
├─ candidates[] (候補ウォッチリスト)
│   ├─ symbol, 追加理由
│   ├─ 最終選定されたか (bool)
│   ├─ 見送り理由
│   ├─ 事後データ (10/50/200本先のOHLC)
│   └─ 自己評価タグ
├─ final_decision
│   ├─ symbol, エントリー有無
│   ├─ エントリー見送り理由(該当時)
│   ├─ 検討したスタイル[] (見送り時も記録可)
│   ├─ 事後データ
│   └─ 自己評価タグ
├─ 中断・再開状態

Trade (エントリー時のみ)
├─ session_id
├─ mode (training | real)
├─ エントリー日時, 方向, 価格, SL, TP
├─ 決済日時, 決済価格, 決済理由
├─ pips損益
├─ 金額損益 (realのみ)
├─ ロット (realのみ)
├─ MT5_order_id (realのみ)
├─ style_id (選択したトレードスタイル)
├─ style_selection_reason (スタイル選定理由)
└─ 後悔指標データ(続き観察期間のOHLC)

TradingStyle (ユーザー定義のトレードスタイル)
├─ id (文字列キー: 'short', 'mid', 'news' 等)
├─ name (表示名)
├─ primary_timeframe
├─ expected_hold_time
├─ expected_rr
├─ typical_sl_pips
├─ description (運用ルール・説明、自由記述)
└─ is_active (有効/無効フラグ)

Scenario (Tradeと1対1)
├─ 環境認識, 相場観
├─ 銘柄選定理由, 見送った候補
├─ 指標認識
├─ 波動カウント
├─ メインシナリオ, 代替シナリオ1, 代替シナリオ2
├─ エントリー根拠
├─ タグ[]
├─ 決済時メモ
└─ 振り返りメモ

PsychologyLog (realモードのTradeに紐づく)
├─ trade_id
├─ phase (entry | holding | exit)
├─ timestamp
├─ 感情タグ (落ち着き/焦り/興奮/恐怖/退屈/その他)
├─ 直近成績影響 (連勝中/連敗後/平常/意識せず)
├─ 身体状態 (良好/疲労/睡眠不足/空腹/その他)
├─ 自由記述メモ
└─ 自己評価 (冷静/感情に押された)

Drawing (Sessionに紐づく描画保存)
├─ session_id
├─ 種類 (line / fibonacci / label / trendline)
├─ 座標データ
└─ ラベル文字

Account (realモード用、MT5から同期)
├─ 残高, 口座通貨, レバレッジ
├─ 証拠金余力
└─ 最終同期時刻

Setting
├─ 対象銘柄リスト
├─ スプレッド設定(銘柄別)
├─ 時間軸プリセット
├─ タグ(固定・自由)
├─ 時間フィルタプリセット
├─ 経済指標表示設定(重要度閾値、表示通貨、シェーディング範囲)
└─ リスク設定(1トレード許容損失%/金額)

EconomicEvent
├─ id, 発表日時
├─ 通貨, 指標名, 重要度(1-3)
├─ 実測値, 予想値, 前回値
└─ サプライズ度(計算値)
```

## 18. 開発優先順位の考え方

- **共通基盤を最優先**: `shared-schema` `market-data` `common-ui-lib` を最初に設計する。後から規約変更すると全アプリに波及するので、初期から丁寧に
- **データ構造を先に固める**: AI分析・リアル連携を見越して、メモ項目・タグ・自己評価・心理状態・modeフラグは最初から揃える
- **market-dataが全ての土台**: データ取得パスがなければ何も始まらない。Phase 1でMT5Provider + キャッシュ層を完成させる
- **アプリBのMVPは「1つのセッションを最後まで回せる」最小構成**: チャート表示、エントリー、決済、基本集計
- **銘柄選定フローは Phase 2 から**: MVP はランダム銘柄直指定で良い
- **アプリDはデータが溜まってから**: Phase 1-3 で1〜2ヶ月運用してデータが蓄積されてから組み込む
- **アプリCは最後**: 実発注を伴うため、他アプリが安定してから着手。デモ口座で十分な検証後に実口座へ
- **発注コードの絶対的隔離**: アプリBには発注関連の依存ライブラリ(MT5パッケージの発注系関数)を含めない。market-dataパッケージも取得機能のみで発注APIは露出させない。誤って将来の改修で混入しないよう、CIで依存関係チェックを入れるのも検討
